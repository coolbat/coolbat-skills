# JTBD

## What problem it solves
JTBD helps uncover the real progress a user is trying to make.
It is best used when feature requests, complaints, or preferences may hide a deeper need.

## Best for
- understanding underlying motivation
- clarifying why users “hire” a product
- uncovering triggers, desired outcomes, and alternatives

## Not for
- scoring feature priority directly
- defining release scope
- replacing metrics

## Use when
- users say they want a feature, but the real reason is unclear
- you need to understand why a product is chosen over alternatives
- you suspect the request is only a surface symptom

## Key questions
- What situation triggered the need?
- What progress is the user trying to make?
- What alternative is the user currently using?
- What outcome does the user care about?

## Output template
When ______,
I want to ______,
so I can ______.

## Common misuse
Do not rewrite a feature request into a fake JTBD sentence and pretend that is insight.
A real JTBD statement must reflect context, motivation, and progress.

## Example
User says: “We need dark mode.”
Possible JTBD:
“When I use the app late at night, I want a visually comfortable interface, so I can continue using it without eye strain.”
