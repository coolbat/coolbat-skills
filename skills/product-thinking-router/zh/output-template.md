# 输出模板

本文档定义 Product Thinking Router 的标准输出结构。

---

## 标准 7 段式输出

每次回答必须按以下结构输出：

### 1. 问题诊断

**格式**：
```
这是一个 [问题类型] 问题。

真正需要做的决策是：[具体决策描述]

[如果信息不足，标注]
ASSUMPTION: 我假设这是 [X] 类问题，基于 [Y] 信号。
```

**要求**：
- 明确问题类型（方向判断/优先级排序/诊断分析/设计规划/指标设计/评估复盘）
- 说明真实决策，而不是表面问法
- 信息不足时明确标注假设

---

### 2. 推荐框架

**格式**：
```
推荐使用以下框架：
- [框架 1]
- [框架 2]（如有）
- [框架 3]（如有）
```

**要求**：
- 最多 3 个框架
- 按使用顺序排列
- 只列名称，不展开解释

---

### 3. 为什么选这些

**格式**：
```
**[框架 1]**：[为什么适用于当前情境]

**[框架 2]**：[为什么适用于当前情境]

**为什么不用 [常见框架 X]**：[说明边界]
```

**要求**：
- 逐个说明适用原因
- 必须说明为什么不选某个常见但不适合的框架
- 解释要具体到当前情境，不要泛泛而谈

---

### 4. 套用到当前问题

**格式**：
```
先回答以下问题：
1. [关键问题 1]
2. [关键问题 2]
3. [关键问题 3]

[或]

按以下步骤分析：
1. [分析步骤 1]
2. [分析步骤 2]
3. [分析步骤 3]
```

**要求**：
- 不要只给框架定义
- 直接把框架应用到用户的具体问题
- 给出具体的分析问题或步骤

---

### 5. 可直接填写的模板

**格式**：

根据框架类型选择合适的模板格式：

**表格型**：
```
| 维度 | 内容 |
|------|------|
| 字段 1 | ______ |
| 字段 2 | ______ |
```

**填空型**：
```
字段 1：______
字段 2：______
字段 3：______
```

**Checklist 型**：
```
- [ ] 检查项 1
- [ ] 检查项 2
- [ ] 检查项 3
```

**要求**：
- 必须可以直接填写
- 字段要具体，不要过于抽象
- 根据框架特点选择合适的格式

---

### 6. 下一步建议

**格式**：
```
下一步建议：

[具体行动 1]

[如果需要，补充]
如果 [条件]，则 [行动 2]
```

**要求**：
- 给出明确的下一步行动
- 行动要可执行，不要模糊
- 必要时给出条件分支

---

### 7. 误用提醒

**格式**：
```
⚠️ 常见误用：[具体描述]

正确做法：[替代方案]
```

**要求**：
- 指出当前场景最容易犯的错误
- 给出正确边界或替代做法
- 要具体到当前情境

---

## 快速模式变体

### 简短模式

触发词：「快速判断」「简短」

**输出结构**：
1. 问题诊断（1 句话）
2. 推荐框架（1 个）
3. 简化模板
4. 下一步建议（1 句话）

---

### 只要模板模式

触发词：「只要模板」「直接给模板」

**输出结构**：
跳过诊断和解释，直接输出：
- 框架名称
- 填写模板
- 填写说明（如需要）

---

### 对比模式

触发词：「对比一下 X 和 Y」

**输出结构**：
```
## [框架 X] vs [框架 Y]

### 适用场景对比
- X 适用于：...
- Y 适用于：...

### 核心差异
| 维度 | X | Y |
|------|---|---|
| 目的 | ... | ... |
| 输入 | ... | ... |
| 输出 | ... | ... |

### 选择建议
如果 [条件 1]，用 X
如果 [条件 2]，用 Y
```

---

## 输出质量检查清单

每次输出前，检查：

- [ ] 是否先诊断问题类型，而不是直接推荐框架？
- [ ] 推荐的框架数量是否 ≤ 3？
- [ ] 是否说明了为什么不选某个常见框架？
- [ ] 是否直接应用到用户的具体问题，而不是只给定义？
- [ ] 模板是否可以直接填写？
- [ ] 下一步建议是否明确可执行？
- [ ] 是否包含误用提醒？
- [ ] 信息不足时是否明确标注假设？

## 示例输出片段

### 好的输出示例

```markdown
### 1. 问题诊断
这是一个"需求真实性 + 优先级"问题。

真正需要做的决策是：判断这个需求是否对应真实用户任务，以及当前是否值得投入资源。

### 2. 推荐框架
- JTBD
- Kano
- RICE

### 3. 为什么选这些
**JTBD**：帮助判断用户提出这个功能背后的真实任务和动机，避免被表面需求误导。

**Kano**：判断这个功能是基础型、性能型还是惊喜型需求，影响优先级判断。

**RICE**：在资源有限时，用结构化方法判断现在该不该做。

**为什么不用 Persona**：Persona 描述用户是谁，但不能直接回答"这个需求是否真实"和"优先级如何"。

### 4. 套用到当前问题
先回答以下问题：
1. 用户在什么场景下提出这个需求？
2. 没有这个功能时，他们现在怎么解决？
3. 这个功能缺失会不会直接导致不满？
4. 它能影响多少用户？投入成本多高？

### 5. 可直接填写的模板
| 维度 | 内容 |
|------|------|
| 触发场景 | ______ |
| 真实任务 | ______ |
| 当前替代方案 | ______ |
| Kano 分类 | ______ |
| Reach | ______ |
| Impact | ______ |
| Confidence | ______ |
| Effort | ______ |
| RICE 得分 | ______ |

### 6. 下一步建议
先收集 5~10 个真实样本，验证这个需求是不是高频任务，再进入 RICE 排序，而不是直接立项。

### 7. 误用提醒
⚠️ 常见误用：把"提及次数多"直接等同于"优先级高"。

正确做法：高频提及不一定等于高价值需求，必须先用 JTBD 判断真实性，再用 RICE 排优先级。
```

---

### 不好的输出示例

```markdown
### 问题分析
这个问题涉及需求管理和产品规划。

### 推荐方法
可以用 JTBD、Kano、RICE、ICE、MoSCoW、Lean Canvas 等方法。

### 方法说明
JTBD 是 Jobs to be Done 的缩写，用于理解用户需求...
Kano 模型由日本学者狩野纪昭提出...

### 建议
建议你先做用户调研，然后分析数据，最后做决策。
```

**问题**：
- 没有明确问题类型
- 推荐了 6 个框架
- 只给定义，没有应用到具体问题
- 没有可填写的模板
- 建议过于模糊
- 没有误用提醒
