# JTBD (Jobs to be Done)

## 一句话定义

JTBD 帮助理解用户真正想要完成的任务和进展，而不是表面的功能需求。

---

## 解决什么问题

当用户提出功能需求、抱怨或偏好时，表面说法往往隐藏了更深层的动机。JTBD 帮助你从"用户说想要什么"转向"用户真正要完成什么"。

**从什么状态到什么状态**：
- 从：被功能请求表象困扰
- 到：理解真实的用户任务、触发情境和期望结果

---

## 适用场景

- **场景 1**：用户提出功能需求，但你怀疑这只是表面症状
- **场景 2**：需要判断竞品功能是否值得跟进
- **场景 3**：想理解用户为什么选择你的产品而不是替代方案

---

## 不适用场景

- **不适用于**：直接做优先级打分（原因：JTBD 帮助理解需求，但不直接给出优先级）
- **不适用于**：定义发布范围（原因：JTBD 关注动机理解，不是范围管理工具）

---

## 输入要求

**必须知道的**：
- 用户在什么情境下提出这个需求
- 用户当前如何解决这个问题（替代方案）

**有助于但非必须的**：
- 用户的具体使用场景描述
- 用户对当前方案的不满点
- 用户期望达成的结果

---

## 输出模板

```
When ______（触发情境）,
I want to ______（想要做的事）,
so I can ______（期望达成的结果）.

---

详细拆解：

触发情境：______
当前替代方案：______
替代方案的不足：______
期望的进展：______
成功的标志：______
```

---

## 常见误用

> ⚠️ 最常见的误用：把功能需求改写成 JTBD 格式，但没有真正理解用户动机
>
> 错误示例：
> - 用户说："我想要深色模式"
> - 错误的 JTBD："When 我使用产品时，I want to 有深色模式，so I can 使用深色模式"
>
> 为什么错误：这只是把需求换了个说法，没有挖掘真实动机
>
> 正确做法：
> - 追问：什么时候想要深色模式？为什么？
> - 可能的真实 JTBD："When 我晚上躺在床上使用产品时，I want to 视觉舒适的界面，so I can 不被强光刺激眼睛，继续使用而不影响睡眠"

---

## 真实案例

**场景**：用户反馈"希望增加导出 PDF 功能"

**输入**：用户在什么情况下需要导出 PDF？

**应用过程**：
1. 追问触发情境：用户说"需要把分析结果发给老板"
2. 追问当前方案：用户说"现在只能截图，但截不全"
3. 追问期望结果：用户说"希望老板能看到完整的数据和图表"

**输出结论**：
```
When 我需要向老板汇报分析结果时,
I want to 完整地呈现数据和图表,
so I can 让老板看到全貌并做出决策.

真实任务：完整呈现分析结果用于汇报
当前替代方案：截图（但截不全）
期望进展：从"部分展示"到"完整呈现"
```

**决策影响**：
- 如果核心任务是"汇报"，可能不只是 PDF，还需要考虑分享链接、演示模式等
- 如果只是"导出"，可能错过了真正的机会

## 工具组合说明

| 搭配工具 | 组合逻辑 |
|---------|---------|
| Kano | 先用 JTBD 理解真实任务，再用 Kano 判断满意度类型 |
| RICE | 先用 JTBD 验证需求真实性，再用 RICE 排优先级 |
| Journey Map | JTBD 聚焦单个任务，Journey Map 看完整流程 |

---

## 关键问题清单

使用 JTBD 时，必须回答：

1. 用户在什么情境下遇到这个问题？
2. 用户现在怎么解决这个问题？
3. 当前方案哪里不够好？
4. 用户真正想要达成什么结果？
5. 如果这个任务完成了，用户的状态会有什么变化？
