# Router Rules — 路由规则

## 目标

将模糊的产品问题转化为正确的分析路径。

---

## Rule 1：识别真实决策

表面问法 ≠ 真实决策。必须先还原用户真正要做的决策。

### 常见转换

| 表面问法 | 真实决策 | 主类别 | 次类别 |
|---------|---------|--------|--------|
| "要不要做这个功能？" | 需求真实性 + 优先级判断 | 方向判断 | 优先级排序 |
| "用户一直提这个需求" | 需求分类 + 投入判断 | 方向判断 | 优先级排序 |
| "竞品上了这个功能" | 跟进必要性判断 | 方向判断 | 优先级排序 |
| "新产品怎么开始？" | 方向梳理 + 假设识别 | 设计规划 | 方向判断 |
| "增长停滞了" | 漏斗断点定位 + 根因分析 | 诊断分析 | 评估复盘 |
| "注册很多但转化差" | 激活环节诊断 | 诊断分析 | 设计规划 |
| "用户留不住" | 留存断点 + 体验摩擦 | 诊断分析 | 评估复盘 |
| "怎么定北极星指标？" | 价值定义 + 指标对齐 | 指标设计 | 方向判断 |
| "团队目标很多" | 指标收敛 + 优先级 | 指标设计 | 优先级排序 |
| "怎么排 roadmap？" | 多选项排序 | 优先级排序 | 设计规划 |
| "MVP 应该保留什么？" | 范围收敛 | 优先级排序 | 设计规划 |
| "上线后表现一般" | 根因诊断 + 复盘 | 评估复盘 | 诊断分析 |
| "为什么用户没用起来？" | 激活失败根因 | 诊断分析 | 评估复盘 |

---

## Rule 2：按问题类别路由

### A. 方向判断

**信号词**：要不要做、值不值得、是否成立、该不该跟竞品、方向对不对

**推荐框架**：
- JTBD（理解真实动机）
- Kano（判断满意度类型）
- Lean Canvas（0→1 方向梳理）
- Assumption Tree（识别关键假设）

**典型组合**：
- 需求判断：JTBD + Kano + RICE
- 新产品方向：Lean Canvas + Assumption Tree + JTBD

---

### B. 优先级排序

**信号词**：先做哪个、怎么排序、资源有限、一堆需求、MVP 保留什么

**推荐框架**：
- RICE（结构化打分排序）
- ICE（快速粗排）
- MoSCoW（范围收敛）

**典型组合**：
- Backlog 排序：RICE
- MVP 缩减：MoSCoW + ICE
- 快速决策：ICE

---

### C. 诊断分析

**信号词**：为什么、根因是什么、增长停了、转化差、留存差、哪里出了问题

**推荐框架**：
- AARRR（增长漏斗诊断）
- 5 Whys（根因追踪）
- Journey Map（体验摩擦定位）
- HEART（体验质量评估）

**典型组合**：
- 增长诊断：AARRR + Journey Map + 5 Whys
- 体验问题：Journey Map + HEART + 5 Whys
- 留存分析：AARRR + Journey Map

---

### D. 设计规划

**信号词**：怎么做、如何拆解、从哪里开始、怎么设计

**推荐框架**：
- Lean Canvas（0→1 结构化）
- Assumption Tree（假设优先级）
- Journey Map（流程设计）
- JTBD（用户视角）

**典型组合**：
- 0→1 启动：Lean Canvas + Assumption Tree
- 流程设计：Journey Map + JTBD

---

### E. 指标设计

**信号词**：看什么指标、北极星怎么定、团队目标太散、怎么衡量成功

**推荐框架**：
- North Star Metric（价值指标对齐）
- HEART（体验指标体系）
- AARRR（增长指标拆解）

**典型组合**：
- 指标体系：North Star + HEART
- 增长指标：North Star + AARRR

---

### F. 评估复盘

**信号词**：上线后、效果一般、为什么没用起来、要复盘什么

**推荐框架**：
- 5 Whys（根因分析）
- HEART（体验评估）
- AARRR（漏斗复盘）

**典型组合**：
- 上线复盘：5 Whys + HEART + AARRR
- 失败分析：5 Whys + Journey Map

---

## Rule 3：考虑产品阶段

### 0→1 阶段

**信号**：新产品、新方向、想法阶段、还没上线

**偏好框架**：
- Lean Canvas
- Assumption Tree
- JTBD
- ICE

**避免框架**：
- HEART（还没有用户体验数据）
- Kano（还没有用户反馈样本）

---

### 早期验证阶段

**信号**：有初始用户、找 PMF、小规模测试

**偏好框架**：
- JTBD
- AARRR
- RICE
- Journey Map

**避免框架**：
- 过于精细的指标体系（数据量不足）

---

### 成长期

**信号**：增长停滞、需要提速、扩大规模

**偏好框架**：
- AARRR
- North Star
- RICE
- HEART

---

### 成熟优化期

**信号**：精细化运营、体验优化、增量改进

**偏好框架**：
- HEART
- Kano
- Journey Map
- 5 Whys

---

## Rule 4：最小工具集规则

### 1 个框架
**适用**：问题类型明确且单一

**示例**：
- "怎么排这 5 个需求的优先级？" → RICE
- "为什么用户流失？" → 5 Whys
- "怎么定北极星指标？" → North Star

---

### 2 个框架
**适用**：需要两个互补视角

**示例**：
- "这个功能要不要做？" → JTBD + Kano
- "增长停了先查哪里？" → AARRR + Journey Map
- "怎么定指标体系？" → North Star + HEART

---

### 3 个框架
**适用**：需要完整链路（诊断 → 决策 → 度量）

**示例**：
- "用户一直提这个需求，要不要做？" → JTBD + Kano + RICE
- "增长停滞了怎么办？" → AARRR + Journey Map + 5 Whys
- "新产品方向太模糊" → Lean Canvas + Assumption Tree + JTBD

---

## Rule 5：始终包含误用警告

每次推荐框架时，必须说明：
1. 为什么不选某个常见但不适合的框架
2. 当前场景最容易犯的错误
3. 正确使用边界

### 常见误用对照

| 框架 | 常见误用 | 正确边界 |
|------|---------|---------|
| JTBD | 把功能需求改写成 JTBD 格式就当作洞察 | 必须反映真实情境、动机和进展 |
| Kano | 单独用 Kano 排优先级 | Kano 只分类满意度，不能替代 ROI 排序 |
| RICE | 信息不足时强行打分 | 信息不足时降低 Confidence 或改用 ICE |
| AARRR | 默认问题在获客环节 | 很多产品问题在激活或留存 |
| North Star | 选最大的数字当北极星 | 必须反映用户价值，不是虚荣指标 |
| Lean Canvas | 填完就当作验证了方向 | Lean Canvas 是假设地图，不是证据 |
| 5 Whys | 把第 5 个 Why 当作绝对真相 | 5 Whys 是思考工具，结论需验证 |

## 快速路由参考卡

| 用户问题 | 第一步框架 | 可选补充 | 避免使用 |
|---------|----------|---------|---------|
| 要不要做这个功能 | JTBD | Kano + RICE | 单独用 Kano |
| 新产品方向模糊 | Lean Canvas | Assumption Tree + JTBD | 直接用 RICE |
| 增长停滞 | AARRR | Journey Map + 5 Whys | 只看获客数据 |
| 一堆需求排序 | RICE | MoSCoW | ICE（信息充足时） |
| 怎么定北极星 | North Star | HEART | 选虚荣指标 |
| 体验差 | Journey Map | HEART + 5 Whys | 只看单点 |
| 上线效果差 | 5 Whys | HEART + AARRR | 停留在表象 |
| MVP 保留什么 | MoSCoW | ICE + Lean Canvas | 什么都保留 |
| 留存差 | AARRR | Journey Map + 5 Whys | 只看留存数字 |
| 竞品跟进判断 | JTBD | Kano + RICE | 直接跟进 |
